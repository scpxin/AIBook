- Date: 2026-07-19

## 一、关键问题 (P0 — 阻塞部署 / 安全风险)

### 1. docker-compose.yml 前端卷挂载路径错误

**文件**: `docker-compose.yml:10`

```yaml
volumes:
  - ./dist:/usr/share/nginx/html          # 错误
  - ./frontend-nginx.conf:/etc/nginx/conf.d/default.conf  # 文件不存在
```

- `./dist` 实际路径是 `./frontend/dist`，当前配置会装载空目录，导致前端 404
- `frontend-nginx.conf` 文件不存在于仓库中，Nginx 无法启动

**修复**: 将路径改为 `./frontend/dist`，并创建 `frontend-nginx.conf` 模板。

---

### 2. SSL 证书验证完全关闭

**文件**: `novel_creator/ai_client.py:10-11`

```python
_ssl_context.check_hostname = False
_ssl_context.verify_mode = ssl.CERT_NONE
```

对所有 AI API 请求均不验证 SSL 证书，存在中间人攻击风险。

**修复**: 改为仅在开发/调试环境关闭，生产环境启用验证：

```python
_ssl_context = ssl.create_default_context()
if os.getenv("AI_VERIFY_SSL", "true").lower() == "false":
    _ssl_context.check_hostname = False
    _ssl_context.verify_mode = ssl.CERT_NONE
```

### 3. Dockerfile 引用不完整的依赖文件

**文件**: `Dockerfile:5`

```dockerfile
COPY requirements.txt .      # 仅 3 个包: fastapi, uvicorn, pydantic
RUN pip install --no-cache-dir -r requirements.txt
```

`novel_creator/` 模块实际无需额外 pip 依赖（标准库 `urllib` 实现），但若后续增加依赖会引入不一致。

另外 `app/requirements.txt` 使用 `uvicorn==0.30.6`（无 `[standard]`），而根目录 `requirements.txt` 用 `uvicorn[standard]==0.30.6`。Dockerfile 使用根目录文件，但 docker-compose 的 `pip install` 命令引用 `app/requirements.txt` —— **两处依赖不一致**。

**修复**: 统一使用 `app/requirements.txt` 或根目录 `requirements.txt`，保持一个真实来源。修改 Dockerfile 为 `COPY app/requirements.txt .`

---

### 4. 零测试覆盖

项目共 4772 行核心 Python 代码（generator: 1500, database_v2: 1641, prompts: 1008, craft_prompts: 623），加前端 Vue/TS 代码，**没有任何单元测试、集成测试或 E2E 测试**。19 模块流水线、AI 生成器、数据库操作等关键逻辑零覆盖。

**建议**: 至少为核心模块添加测试：
- `novel_creator/database_v2.py` → SQLite 集成测试
- `novel_creator/generator.py` → Mock AI 响应测试
- `app/services/pipeline.py` → 状态机测试
- 前端 → 组件渲染 + API mock 测试

## 二、高风险问题 (P1 — 影响可维护性和可靠性)

### 5. 硬编码内网 IP 地址

| 位置 | 内容 |
|------|------|
| `docker-compose.yml:37` | `http://101.35.133.34:5000/...` (CONTENT_API) |
| `app/main.py:37` | `http://140.143.210.177` (CORS ALLOWED_ORIGINS) |
| `README.md:213` | `140.143.210.177` (部署文档) |
| `.monkeycode/MEMORY.md:39` | `140.143.210.177` (服务器记录) |

- README 和 MEMORY.md 中属于正常文档引用
- `docker-compose.yml` 和 `main.py` 中的 IP 应改为环境变量或配置文件

**修复**: docker-compose 中 CONTENT_API 和 CORS 地址统一走环境变量，提供 `.env.example` 占位。

---

### 6. 日志重复配置

**文件**: `novel_creator/generator.py:10`, `novel_creator/prompts.py:10`

两个顶层模块各调用一次 `logging.basicConfig()`，写入同一个日志文件。`main.py:26` 还有第三次配置。

**问题**: `logging.basicConfig()` 第二次调用不生效，会导致 generator.py 的 handler（FileHandler）和 main.py 的 handler（StreamHandler + FileHandler）冲突，实际行为不可预测。

**修复**: 仅在 `main.py` 入口配置一次日志，其他模块用 `logging.getLogger(__name__)` 获取 logger。

---

### 7. `.env.production` 被提交但已被 `.gitignore`

`.gitignore:6` 声明 `*.env.production` 不跟踪，但 `frontend/.env.production` 已存在于仓库中。这会导致修改该文件后 Git 不跟踪变更。

**修复**: 从 Git 中移除该文件：`git rm --cached frontend/.env.production`，改用 `frontend/.env.production.example` 作为模板。

---

### 8. 缺少代码质量工具

项目没有任何 lint/format 配置：
- Python: 无 `pyproject.toml`, `.flake8`, `.pylintrc`
- 前端: 无 `.eslintrc`, `.prettierrc`

## 三、中等优先级 (P2 — 技术债)

### 9. urllib 替代 httpx

AI 客户端使用标准库 `urllib` 进行 HTTP 请求，无法复用连接，在 `--workers 2` 多进程模式下每个请求都需要新建 SSL 握手。

**建议**: 换用 `httpx`（支持 async，连接池复用），与 FastAPI 生态一致。

---

### 10. V1/V2 双数据库架构

`novel_creator/database.py`（V1，771 行）和 `novel_creator/database_v2.py`（V2，1641 行）各自管理独立的数据库连接和表结构，增加了维护负担。

**建议**: 若 V1 表已无引用，可标记废弃；若有引用，应将 V1 表迁移至 V2 统一管理。

---

### 11. MEMORY.md 路径过期

多处引用 `/workspace/deploy/` 目录，但当前代码直接在 `/workspace/` 根目录下，说明项目目录结构已重组。

**建议**: 更新 MEMORY.md 中的路径引用，或确认是否需要将代码迁回 `deploy/` 目录。

---

### 12. docker-compose.yml `version` 字段已废弃

Docker Compose V2 不再需要 `version` 字段，当前 `version: "3.8"` 仅为历史遗留。移除不影响功能。
